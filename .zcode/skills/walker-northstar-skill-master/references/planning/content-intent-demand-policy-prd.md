# 内容创作意图与需求关系 PRD

## Mode

Design Policy：本文定义哪些内容只服务个人表达，哪些内容可以根据用户需求创作，哪些内容可以进入 Agent / Skill 调用。

## Core Principle

内容不是单一产品。有些内容只对 Walker 自己负责，有些内容面向用户问题，有些内容沉淀为方法，有些内容才有资格进入 Skill。

## Content Types

| 内容类型 | 是否看用户需求 | 是否可公开 | 是否可进入 Agent |
|---|---|---|---|
| 个人思考 | 不看用户需求 | 由站主决定 | 默认只作背景，除非明确开放 |
| 生活记录 | 不看用户需求 | 由站主决定 | 默认不进入任务执行 |
| 工具笔记 | 可参考用户需求 | 可公开 | 可进入推荐和解释 |
| 教程文章 | 主要看用户需求 | 可公开 | 可进入答案匹配 |
| 方法卡 | 来自反复有效经验 | 可公开或私密 | 可进入 Agent 判断 |
| Skill 候选 | 必须经过验证 | 由站主决定 | 仅候选，不直接调用 |
| 正式 Skill | 必须稳定复现 | 可公开或私有 | 可被 Agent 调用 |

## Core Modules

| 模块 | 一句话职责 |
|---|---|
| 内容意图标注模块 | 标明一篇内容是自我表达、教程、工具说明、方法沉淀还是 Skill 候选。 |
| 需求来源关联模块 | 记录某篇需求型内容来自哪些真实问题、直播反馈或对话样本。 |
| 公开边界模块 | 决定内容是否 public、draft、private 或 admin-only。 |
| AI 使用边界模块 | 决定内容只能作背景、可以引用、可以推荐，还是可以执行任务。 |
| 内容更新版本模块 | 记录一篇内容的连续修订和版本关系。 |
| 创作决策模块 | 判断某个需求是否值得写成文章、教程、直播话题、工具页或 Skill 候选。 |

## Module Relationships

| 关系 | 方向 | 理由 |
|---|---|---|
| 调用 | 内容意图标注模块 -> 公开边界模块 | 内容意图不同，公开风险不同。 |
| 调用 | 内容意图标注模块 -> AI 使用边界模块 | 不是所有公开内容都适合被 Agent 执行调用。 |
| 触发 | 需求事件 -> 需求来源关联模块 | 用户真实需求可以成为内容创作来源。 |
| 调用 | 需求来源关联模块 -> 创作决策模块 | 需要判断需求频率、价值、边界和站主是否愿意回应。 |
| 调用 | 创作决策模块 -> 内容意图标注模块 | 决定要写什么类型的内容。 |
| 调用 | 内容更新版本模块 -> 需求来源关联模块 | 内容更新后，应能追溯它解决了哪些需求。 |

## Decision Rules

- “于是我辞职 / 我思之思”等个人思考，不为了迎合需求而写。
- 常用工具、教程、方法解释，可以根据用户需求创作。
- 直播、某鱼、评论、AI 对话中的真实问题，可以进入需求池，但必须脱敏。
- 用户需求只能提供选题信号，不能替代站主判断。
- 私密内容、未整理内容、脆弱表达，默认不进入公开推荐。
- 一篇内容可以多次更新，版本关系应被记录，而不是把旧思考抹掉。

## Acceptance Criteria

- [ ] 每篇内容能标明主要创作意图。
- [ ] 需求型内容能追溯来源需求。
- [ ] 个人表达内容不会被误当成用户答案资产。
- [ ] Agent 推荐内容前能读取 AI 使用边界。
- [ ] 内容更新能保留版本关系。
