# 原始经验采集与验证系统 PRD

## Mode

Design PRD：本文定义“先积累事实，再提炼规则”的产品模块、职责边界和模块关系。

## Target Outcome

不要一开始就假设用户真正需要什么。先保留真实事件、原话、场景、行动和结果，再从多次实践中提炼需求模式、成功标准、方法卡和 Skill 候选。

核心路径：

```text
记录真实事件
-> 保存原话
-> 标注场景
-> 记录初步判断
-> 记录帮助动作
-> 收集反馈结果
-> 复盘修正
-> 识别需求模式
-> 提炼成功标准
-> 生成 Skill 候选
```

## Core Modules

| 模块 | 一句话职责 |
|---|---|
| 原始事件采集模块 | 记录直播、某鱼、网站、AI 对话、教程反馈等真实发生的用户事件。 |
| 原话保真模块 | 保存用户原始表达，避免过早总结导致需求失真。 |
| 场景标注模块 | 标注事件来源、用户身份、发生场景、表层需求和上下文条件。 |
| 弱假设记录模块 | 记录当时对用户真实需求的初步判断，并明确这是可推翻假设。 |
| 帮助动作记录模块 | 记录当时给出的文章、方法、工具、Skill、教程或人工建议。 |
| 反馈结果模块 | 记录用户反应和后续结果，包括成功、失败、无反馈、待观察。 |
| 复盘修正模块 | 对比初始判断和最终结果，修正需求理解、方法有效性和失败原因。 |
| 需求模式分析模块 | 从多条真实事件中发现反复出现的问题、需求、失败模式和成功条件。 |
| 成功标准提炼模块 | 从已验证事件中提炼“什么算帮到了用户”的可观察标准。 |
| 实践样本池模块 | 汇总已记录、已反馈、已复盘的真实事件，作为认识和提炼规律的事实基础。 |
| 矛盾识别模块 | 从用户原话、场景、行动和结果中识别表层矛盾、真实矛盾和潜在矛盾。 |
| 主要矛盾判断模块 | 判断当前事件中最决定行动结果的关键矛盾，避免被表层需求带偏。 |
| 条件边界模块 | 记录方法成立所需的身份、场景、资源、时机、技术条件和外部约束。 |
| 失败边界模块 | 记录方法在哪些输入、场景或条件下失效，用于定义 Skill 的不可求解范围。 |
| 规律候选模块 | 将已抽象经验提升为“可能的规律”，但保留待验证状态。 |
| 再实践验证模块 | 将规律候选重新用于新事件，检验它是否能继续帮助用户。 |
| 反例管理模块 | 保存推翻原判断的事件，用反例修正定义域、方法边界和成功标准。 |
| 方法成熟度评估模块 | 判断经验处于素材、方法雏形、稳定方法、Skill 候选还是正式 Skill。 |

## Module Relationships

| 关系 | 方向 | 理由 |
|---|---|---|
| 触发 | 管理员复盘工作台模块 -> 原始事件采集模块 | 站主在直播、某鱼、网站或教程反馈后，手动记录一条真实事件。 |
| 调用 | 原始事件采集模块 -> 原话保真模块 | 事件进入系统时，必须先保存用户原话。 |
| 调用 | 原始事件采集模块 -> 场景标注模块 | 原话需要绑定来源、身份、场景和上下文。 |
| 调用 | 场景标注模块 -> 弱假设记录模块 | 标注完成后，记录当时的初步判断。 |
| 调用 | 弱假设记录模块 -> 帮助动作记录模块 | 有了判断后，记录当时给了什么帮助。 |
| 触发 | 用户后续反馈 -> 反馈结果模块 | 用户认可、追问、行动、沉默、失败或成功，都应成为事件结果。 |
| 调用 | 反馈结果模块 -> 复盘修正模块 | 结果出现后，必须回看最初判断是否正确。 |
| 调用 | 复盘修正模块 -> 实践样本池模块 | 复盘后的事件进入可分析样本池。 |
| 调用 | 实践样本池模块 -> 矛盾识别模块 | 基于真实样本识别用户问题中的矛盾结构。 |
| 调用 | 矛盾识别模块 -> 主要矛盾判断模块 | 多个矛盾出现后，需要判断哪个最决定结果。 |
| 调用 | 主要矛盾判断模块 -> 条件边界模块 | 判断主要矛盾时必须记录成立条件。 |
| 调用 | 条件边界模块 -> 失败边界模块 | 条件不满足时形成失败边界。 |
| 调用 | 主要矛盾判断模块 -> 需求模式分析模块 | 多次相似矛盾和有效处理方式，才可以抽象为模式。 |
| 调用 | 需求模式分析模块 -> 成功标准提炼模块 | 模式需要进一步沉淀为可观察的成功标准。 |
| 触发 | 规律候选模块 -> 再实践验证模块 | 规律候选不能直接成为规则，必须回到新事件中继续验证。 |
| 调用 | 再实践验证模块 -> 反例管理模块 | 新事件推翻原规律时，必须记录反例。 |
| 调用 | 方法成熟度评估模块 -> Skill 候选生成模块 | 达到成熟度的经验才进入 Skill 候选。 |

## First MVP Loop

第一版不要先做复杂 Agent，也不要先做自动 Skill 生成。先做这个事实闭环：

```text
记录一条真实事件
-> 保存用户原话
-> 标注身份 / 场景 / 表层需求
-> 记录初步判断
-> 记录给出的帮助
-> 记录用户反馈和结果
-> 复盘判断是否修正
-> 人工标记是否出现需求模式
```

当积累 20-50 条真实事件后，再进入下一层：

```text
需求模式分析
-> 成功标准提炼
-> 方法卡沉淀
-> Skill 候选生成
-> Skill 准入判断
```

## Acceptance Criteria

- [ ] 每条事件能保留用户原话。
- [ ] 每条事件能标注来源、场景、身份和表层需求。
- [ ] 每条事件能记录当时判断和帮助动作。
- [ ] 每条事件能记录反馈结果。
- [ ] 复盘时能对比“初始判断”和“最终结果”。
- [ ] 多条事件能被聚合为需求模式。
- [ ] 失败案例能作为反例修正边界。
