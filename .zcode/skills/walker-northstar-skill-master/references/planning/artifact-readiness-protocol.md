# Artifact Readiness Protocol

Generated: 2026-06-11

Stage: AdgaiWalker 个人系统 / 发布接口桥接

## 目的

本协议取代固定线性阶段提醒。`walker-northstar` 不应把开发流程写死为 `Idea -> PRD -> Architecture -> Plan -> Todo -> Dev`，而应围绕一个核心文档判断：

```text
这个文件现在缺什么？
它是否已经足够支撑下一步行动？
下一步行动前，Walker 需要确认什么？
```

核心原则：

```text
需求决定一件事是否存在。
文档成熟度决定下一步能否行动。
Skill 不替 Walker 做决定，只提醒 Walker 该确认什么。
```

## 核心对象

每一轮只追踪一个核心文档。

核心文档可以是：

- PRD
- 架构设计
- 计划
- todo
- 复盘
- Skill 协议
- 其他正在承载本轮判断的文档

但在 `walker-northstar` 中，优先把 PRD 理解为母文档：

```text
PRD 不只是“需求阶段”的文件。
PRD 可以承载需求、边界、模块、架构、计划、验收和复盘入口。
```

## 文档成熟度层

### 1. 存在性

要回答：

- 这件事为什么要存在？
- 如果不做，会损失什么？
- 它服务的是自用、公开回答、需求收集、skill 形成、发布接口，还是 NorthStar？

不通过信号：

- 只是想做，但说不清解决什么问题。
- 只是因为技术上能做。
- 把社区复杂度倒灌进个人系统。

提醒：

```text
当前文档还没有证明这件事为什么存在。是否先补“存在性”？
```

### 2. 需求性

要回答：

- 谁需要？
- 在什么场景需要？
- 当前痛点、期待和差距是什么？
- 需求来自 Walker 自用、真实用户、内容创作、工具实践，还是长期社区愿景？

不通过信号：

- 用户不清楚。
- 需求和解决方案混在一起。
- 把“我能做什么”误当成“别人需要什么”。

提醒：

```text
当前文档已有方向，但需求对象不清。是否先补用户、场景和问题？
```

### 3. 边界性

要回答：

- 本轮做什么？
- 本轮不做什么？
- 哪些是个人系统，哪些是发布接口，哪些属于未来社区？
- 哪些内容涉及隐私、权限、公开边界？

不通过信号：

- 范围越写越大。
- 账号、社区、推荐、审核等复杂度提前进入个人系统。
- 私密内容和公开内容没有分界。

提醒：

```text
当前文档需要先收边界。是否先明确做什么和不做什么？
```

### 4. 结构性

要回答：

- 核心模块有哪些？
- 每个模块一句话职责是什么？
- 模块之间的依赖、调用、触发、实现关系是什么？
- 是否依赖接口而不是依赖具体实现？

不通过信号：

- 只有功能列表，没有模块关系。
- 架构图好看，但不能指导实现。
- 数据边界和权限边界不清。

提醒：

```text
当前文档已具备需求和边界，但结构不够清楚。是否补模块与关系？
```

### 5. 执行性

要回答：

- 先做什么，后做什么？
- 哪些动作属于 Skill 仓库？
- 哪些动作属于产品仓库？
- 哪些动作必须先经过 Human Gate？

不通过信号：

- 计划不能拆成可执行动作。
- todo 混入未确认需求。
- 蓝图层和现实层变更混在一起。

提醒：

```text
当前文档可以进入执行拆解。是否生成 plan/todo？
```

### 6. 验收性

要回答：

- 怎么证明它做成了？
- 测试、构建、页面体验、接口返回、权限边界、文档对账分别看什么？
- 失败如何记录？

不通过信号：

- 只有“完成了”的主观判断。
- 没有运行、测试或人工体验证据。
- 没有记录失败假设。

提醒：

```text
当前文档缺验收标准。是否补可验证结果？
```

### 7. 复盘性

要回答：

- 本轮发生了什么？
- 哪些假设被证伪？
- 哪些提醒有用，哪些多余？
- 是否形成 case、pattern 或 rule 晋升候选？

不通过信号：

- 做完即结束，没有沉淀。
- 所有经验直接写进 SKILL.md。
- 没有把现实结果回写到 cycle ledger。

提醒：

```text
当前轮次可以复盘。是否生成 case / pattern 判断并更新 cycle ledger？
```

## 下一步判断方式

每次用户说“继续”“开始”“下一步”“现在做到哪了”，按以下顺序判断：

```text
1. 当前核心文档是哪一个？
2. 它已经满足哪些成熟度层？
3. 它现在最缺哪一层？
4. 补这一层是否需要 Walker 确认？
5. 补完后是否足以进入现实层行动？
```

输出格式：

```text
Stage:
核心文档：
当前成熟度：
已具备：
当前缺口：
建议下一步：
需要 Walker 确认：
```

## 不做

- 不把流程写死为单一路径。
- 不要求每个任务都经历所有层。
- 不把架构设计从 PRD 母文档中硬拆出去。
- 不把一次偶然经验直接晋升为 Skill 规则。
- 不因为文档成熟就自动执行高风险动作。
