## 一、一句话

盘根让信息可流通、可沉淀、可更新，连接人与人之间的信任。

---

## 一·附 产品本质

盘根做一件事：**让校园信息从混乱变成有序。**

四个维度，缺一不可：

```
信息维度（事的层面）：
  可流通 — 信息从存在的地方到达需要的人
  可沉淀 — 信息留下来，不被时间冲走
  可更新 — 信息保持活着，变化了有人知道

人的维度（人的层面）：
  信任   — 连接人与人，让知识的流动有人驱动
```

前三个是信息的生命周期。第四个是人的连接。

没有信任，前三个是死循环：
- 信息流通了但没人信 → 没人来看
- 信息沉淀了但没人维护 → 变成死档案
- 信息需要更新但没人反馈 → 变成过期信息

信任让系统活过来：

```
读者信任作者 → 因为他的内容被 89 人确认过
作者信任读者 → 因为有人认真反馈"有变化"而不是骂人
作者信任平台 → 因为他的内容不会消失，他的名字挂在上面
平台信任作者 → 因为认证机制保证了质量门槛
```

四个信任关系，缺一个系统就塌。

而盘根卖的最终不是信息，是信任。学生打开盘根，不是"搜一个信息"，是"找一个可信的答案"。信息到处都有。可信的信息没有。

## 二、为什么

学生有两种信息需求，现在都没有好的解决方案：

**即时需求：** "食堂几点关门？" → 微信群里问，等 20 分钟有人回，答案可能过期了
**认知需求：** "我想了解选课" → 翻 10 个群、5 个公众号、问 3 个学长，信息碎片化，矛盾，过时

微信群解决了即时需求的一部分，但解决不了认知需求。群里的信息是散的——每次有人问同样的问题，每次重新回答。没人把关于"食堂"的所有信息整理成一个完整的、持续更新的体系。第 100 个人问同样的问题时，和第 1 个人问的时候一样混乱。

盘根存在的四个原因：

1. **把已经存在但不可达的信息变得可达** — 搜一下就有了
2. **把碎片信息组织成有体系的知识** — 关于一个话题的所有信息，有人整理、有人维护、有人确认
3. **让同一个问题只需要被回答一次** — 系统记录哪些问题被反复问，自动驱动内容生产填补空白，消除重复
4. **让 AI 服务人** — 学生不需要学习 AI，只需要用盘根，AI 就在为他工作

第一个是搜索的价值。第二个是空间的价值。第三个是数据驱动的价值。第四个是 AI 赋能的价值。

### 2.1 AI 服务人，不是人学 AI

盘根校园站有一个隐性任务：推动校园 AI 化。

但方式不是"教学生用 AI"。方式是**让学生在解决日常问题的过程中，自然地、无感地使用 AI，体验 AI 为自己工作的价值。**

```
传统思路：先学 AI → 再用 AI 解决问题
盘根思路：先用盘根解决问题 → 不知不觉中 AI 在为你工作 → 有的人好奇了 → 主动学 AI
```

公理一·人类主权：人不应该为了用 AI 而用 AI。AI 应该在人类需要的时候出现，不需要的时候消失。盘根就是这个原则的实践——学生搜食堂信息，他不需要知道背后是关键词匹配还是 AI 生成。他只需要得到答案。

### 2.2 从用到学的路径

校园站和全球站的关系，不是"两个独立产品"，是**同一条路上的两个阶段**：

```
阶段 1（校园站）：学生在用 AI
  → 搜"食堂" → AI 在后台帮他找到最相关的信息
  → 搜"选课" → 没有本地答案 → AI 给出参考答案
  → 写帖子 → AI 帮他自动分类和推荐标签
  → 学生感受到："这个东西很好用，搜什么都有"

阶段 2（自然转化）：有学生开始好奇
  → "这个搜索怎么这么准？"
  → "它怎么知道我关注麻辣烫？"
  → AI 回答卡片上的"由 AI 生成"标签
  → 学生开始意识到：AI 在帮我

阶段 3（全球站）：主动学习 AI
  → 点击"了解更多 AI 工具" → 进入全球站
  → 学习 AI 写作、AI 绘画、AI 编程
  → 从被动使用到主动掌握
```

**校园站是 AI 的入口。全球站是 AI 的进阶。**

不是通过教育让人接受 AI，是通过体验让人理解 AI。先甜后知——先享受 AI 带来的便利，再好奇背后的原理。

## 三、核心体验

盘根提供两种价值，对应两种用户行为：

```
问 — 搜一个问题，3 秒得到可信答案（工具价值）
看 — 进一个空间，看到关于某个话题的完整知识地图（认知价值）
```

### 3.1 搜索：即时答案

```
打开盘根 → 输入问题 → 得到答案

本地有答案：展示，标注确认时间和确认人数
本地没有：AI 先答，标注"AI 生成"，引导人类补充真实信息
```

这是工具价值。学生有个具体问题，搜完就走。

### 3.2 空间：有体系的知识

```
进入"三食堂"空间 → 看到：

  三食堂有 8 个窗口，排队最短的是 4 号（约 3 分钟）     ← AI 实时摘要

  📄 食堂完整测评（89 人确认）                          ← 深度文章
  📄 各窗口价格对比（确认于 5 天前）                     ← 结构化数据
  💬 烤冷面新窗口真不错（3 分钟前）                      ← 最新动态
  💬 麻辣烫搬了（2 小时前）                              ← 变化信息
  📄 推荐菜品排行（42 人确认）                          ← 众包排名
```

这是认知价值。学生不是在找一个答案，是在了解一个话题。空间把碎片化的信息组织成有层次、有结构、有人维护的知识体系。

**微信群给你一个答案。盘根给你一个认知地图。**

### 3.3 两种价值的互补

搜索和空间不是两个独立功能，是一个循环：

```
学生搜"食堂" → 得到即时答案 → 答案来自"三食堂"空间
  → 学生点进去 → 看到完整知识地图 → 建立认知
  → 学生有问题 → 在空间里问 → 帖子自动归属这个空间
  → 新信息补充到知识体系中 → 搜索结果变得更好
```

搜索是入口。空间是沉淀。信息从搜索流向空间，又从空间反哺搜索。

## 四、用户只需知道三件事

盘根对用户只有三个概念。不是四个，不是五个。

```
问 — 搜一个问题，得到一个答案
看 — 进一个空间，看到关于某个话题的完整知识
写 — 写点什么，成为知识体系的一部分
```

系统内部有等级、有标签、有排序算法、有审核队列。用户不需要知道这些的存在。

复杂是系统的事。简单是用户的事。

### 为什么"空间"是核心概念

PRD 说"做减法"。那为什么保留"空间"这个概念？

因为学生确实需要一个地方来**理解一个话题的全貌**。这不是我们强加的概念，是用户的真实需求：

- 大一新生想了解"报到"：不是问一个问题就完了，是需要知道报到流程、宿舍分配、军训、选课、办卡——一整套
- 想了解"三食堂"：不是只知道几点关门，是需要知道每个窗口、价格、排队时间、最新变化——一个完整地图
- 想了解"选课"：不是问一门课好不好，是需要知道必修怎么选、选修推荐什么、避雷哪些——一套体系

空间就是这些知识的容器。它让碎片信息变成有结构的知识。没有空间，盘根只是一个更好的搜索引擎。有空间，盘根是一个活的、有人维护的校园知识网络。

### 搜索框下面那三行是什么

> **v9.0 更新**：推荐形式从三行文本改为 4 张推荐卡片（见第二十六节）。推荐逻辑不变（AI 根据时间/热度实时计算），展示形式更新。

系统根据**时间、用户历史、信息时效性**实时计算当下最相关的 4 个空间，展示为推荐卡片。每张卡片显示空间名和确认数。

- 中午 12 点：食堂信息权重最高
- 晚上 9 点：图书馆和自习室权重最高
- 开学季：报到和选课权重最高
- 考试周：复习资料和自习室权重最高

AI 在后台持续运算，用户只看到结果。

### 搜索行为

用户输入问题后，系统自动处理，不暴露任何技术概念：

```
用户输入 → 关键词匹配本地内容
           → 有结果：展示，标注确认时间
           → 无结果：AI 回答，标注"AI 生成，仅供参考"
```

没有"普通搜索"和"AI 搜索"的切换。没有模式按钮。用户只管问。

AI 回答的底部永远有一个按钮：**"我知道准确信息，来补充"**。引导人类覆盖 AI 的回答。公理一·人类主权：AI 先答，人类裁决。

### 流（Feed）

搜索框下方是按活动排序的混合流。帖子、文章更新、新发布的内容混在一起，不区分类型。

排序依据（用户不可见）：

- 最近有人阅读、确认、收藏、更新的内容浮上来
- 长期无人互动的内容自然沉下去
- 不需要"推到最新"按钮。阅读本身就是推

流中不显示收藏按钮。收藏是详情页的行为，不是浏览时的行为。

### 帖子预览

点击流中的短内容，弹出居中预览浮层，不跳转页面。浮层展示完整内容、图片、标签、回复数。底部有"查看全部回复"按钮，点击才进入完整详情页。

三种关闭方式：点 X、点遮罩、按 Escape。降低浏览成本，快速扫过后再决定是否深入。

## 六、内容空间

### 空间是什么

空间是一个话题下所有知识的容器。包含深度文章、最新动态、实时讨论，混合在一个流里。

不是"知识库列表页"。不是"版块"。不是"频道"。是关于某个话题的**完整认知地图**。

### 空间的核心特征：有体系

空间和微信群的信息有什么区别？**体系感。**

```
微信群里关于食堂的信息：
  · "今天一食堂好吃"（3 月 12 日）
  · "麻辣烫搬了"（不知道几号）
  · "三食堂排队好长"（不知道哪个窗口）
  散乱的、不可检索的、没结构的

盘根"三食堂"空间里的信息：
  📄 食堂完整测评 — 覆盖所有窗口的系统性评测（89 人确认）
  📄 各窗口价格对比 — 结构化数据表格（确认于 5 天前）
  📄 推荐菜品排行 — 众包排名（42 人确认）
  💬 麻辣烫搬了 — 最新变化（2 小时前，有人确认）
  💬 烤冷面新窗口 — 实时动态（3 分钟前）
  有深度的、可检索的、有结构的、有人维护的
```

### 空间页面

```
┌──────────────────────────────────┐
│  ←  三食堂                       │
│                                  │
│  三食堂有 8 个窗口，              │
│  排队最短的是 4 号窗口（约 3 分钟）│
│  · 12 篇内容 · 89 人确认有帮助      │

│                                  │
│  📄 完整测评 · 更新于昨天         │
│     覆盖全部 8 个窗口的味道、分量、│
│     人均价格、推荐指数            │
│     89 人确认有帮助                 │
│                                  │
│  📄 价格对比表 · 确认于 5 天前    │
│  📄 推荐菜品排行 · 42 人确认      │
│                                  │
│  ── 最新动态 ────────────────── │
│                                  │
│  💬 烤冷面新窗口真不错 · 3分钟前  │
│  💬 麻辣烫搬了 · 2小时前         │
│  💬 今天糖醋排骨好吃 · 4小时前    │
│                                  │
│  ┌────────────────────────────┐  │

```

**知识体系在前，最新动态在后。** 这是空间和 Feed 的本质区别。Feed 按时间展示，空间按认知深度展示。

长内容和短内容混合在一个流里，但有视觉分层：深度文章卡片更大、信息更全；动态帖子更紧凑。

没有"文章"和"讨论"的 Tab 切换。用户关心的是"关于这个话题的一切"，不是内容的类型。

### 空间的维护者

每个空间有一个维护者（通常是创建者）。维护者负责：

- 保持内容准确（响应"有变化"反馈）
- 补充新内容（当信息缺失时）
- 组织知识结构（决定文章的排列顺序）

维护者不是版主。没有管理权限。维护者是一个**对某个话题最了解的人**，他的价值在于持续投入和深度认知。

维护者长期不活跃（90 天）时，系统通知资深用户认领。

### 内容空间列表

首页的导航里有"探索"入口，展示所有内容空间的网格。用户可以浏览、搜索、进入任何空间。

每个空间卡片显示：名称、一句话描述、最近活动时间、确认人数。

## 七、阅读页

点击一篇文章进入阅读页。

### 桌面端：三栏

```

│   价格    │ · 推荐   │  ## 概述                    │
│ · 推荐    │ · 时间   │  二食堂麻辣烫自 2024 年...   │
│          │          │                             │
│          │          │  ## 价格                     │
│          │          │  人均 12-15 元...            │
│          │          │                             │
│          │          │  ── 这篇内容对你有帮助吗？──  │
│          │          │  [ 有帮助 ]  [ 有变化 ]       │
│          │          │                             │
│          │          │  ← 上一篇    下一篇 →        │
└──────────┴──────────┴────────────────────────────┘
```

左栏：同一内容空间下的所有文章列表，当前阅读的高亮。
中栏：当前文章的章节目录，跟随滚动高亮当前位置。
右栏：文章正文。

### 移动端：单栏 + 折叠目录

顶部显示可折叠的目录。正文占满宽度。底部是反馈按钮和上下篇导航。

### 文章底部

两个按钮：

- **有帮助** — 确认信号，刷新"上次确认"时间
- **有变化** — 弹出一行输入框"什么变了？"，提交后通知作者

没有评分。没有评论。没有三选一。就像日常对话——"有用"或者"不对了，我告诉你新的"。

每篇文章显示"上次确认于 X 天前"。这是事实，不是评价。读者自己判断可信度。

不使用颜色标签（绿黄红）。原因：颜色标签是公开的成绩单，伤害信任链。

### "有变化"是公共信息

当有人反馈"有变化"时，这个信息不应只发给作者。其他读者也需要知道。

**展示方式：** 文章标题下方显示"有人反馈可能有变化"，点击可查看变化内容。这个标注会一直存在，直到作者更新文章。

原因：公理二·熵减。"有变化"是降低熵的信息——它告诉读者"这篇文章的部分内容可能不准确"。如果这个信息只有作者能看到，其他读者会被误导。信息透明是信任的基础。

## 八、写

### 发短内容

任何注册用户都能发。一个输入框，写完直接发。

系统根据内容自动归类到对应的内容空间。用户可以修改系统推荐的空间。

系统根据内容长度和结构自动判断是"帖子"还是"文章"。用户不需要选。

标签是可选的。系统根据内容自动推荐标签，用户可修改。核心维度：`问` `说` `卖`。

- 问 = 求助、提问
- 说 = 分享、讨论、活动
- 卖 = 二手、交易

### 写长内容

需要认证。不是申请制，是邀请制。

当用户满足条件时，系统推送通知："你现在可以写长文章了，要不要试试？"

认证条件基于活跃度和内容质量，具体阈值后台可配置。初始参考值：
- 注册 ≥ 7 天
- 发帖 ≥ 3 条
- 收到 ≥ 5 次"有帮助"

上线后根据实际数据调整阈值。

用户点"好"，自动获得写作权限，同时系统自动为用户创建一个个人空间（"XX 的地盘"）。

公理一·人类主权：升级是行为自然积累的结果，不是向系统申请来的。

### AI 对话式写作

写长内容时，AI 通过对话帮助用户组织思路：

1. AI 问："你想分享什么？"用户自由表达
2. AI 追问补充细节（"价格是人均还是一份？"）
3. AI 排列内容结构，用户可调整顺序、删除、补充
4. 用户确认后发布

AI 不代笔。AI 不替用户决定什么重要。内容的主人永远是用户。

## 九、信任

盘根卖的最终不是信息，是信任。

信息到处都有。可信的信息没有。学生打开盘根，不是"搜一个信息"，是"找一个可信的答案"。

### 信任的四条线

盘根的信任不是一个点，是四条线，连接四种关系：

```
读者 → 作者：我信你写的东西
  因为：89 人确认过、3 天前刚更新、时间透明

作者 → 读者：我信你会认真读、认真反馈
  因为：反馈机制是"有帮助/有变化"，不是点赞踩，是建设性的

作者 → 平台：我信我的内容在这里有价值、不会被埋没
  因为：内容归属空间、名字挂在上面、影响力数据可见

平台 → 作者：我们信你能保证质量
  因为：认证机制是邀请制、不是谁都能写长文章
```

### 信任飞轮

```
信息准确 → 读者信任 → 愿意来问 → 有人回答 → 信息保持准确
    ↑                                        │
    └────────────────────────────────────────┘
```

飞轮转起来的标志：有人主动写了第二篇内容。不是被邀请写的，不是被奖励写的，是他自己想写的——因为他信他的内容会被看见、被确认、被需要。

### 信任的脆弱性

信任是这四条线中最容易断的。一旦断了，很难修复：

```
破坏读者→作者信任的方式：
  · 信息不准且长期没人更新 → 读者再也不信这个作者
  · 作者抄袭或编造 → 读者再也不信这个平台

破坏作者→读者信任的方式：
  · 反馈被滥用（恶意"有变化"）→ 作者不想再维护
  · 无人阅读无人确认 → 作者觉得没意义

破坏作者→平台信任的方式：
  · 内容被删除无解释 → 作者觉得不安全
  · 排序不透明 → 作者觉得不公平

破坏平台→作者信任的方式：
  · 降低认证门槛涌入低质量内容 → 平台不再可信
```

**红线（第十七节）的本质：保护这四条信任线。** 任何破坏信任的事，无论对增长多有吸引力，都不能做。

### 自然生长

系统内部维护完整的信任等级（游客 → 注册 → 活跃 → 认证 → 资深 → 管理员），但用户端不暴露等级数字、升级进度条、等级名称。

用户感知到的是自然事件：

| 系统等级 | 用户感知 |
|---------|---------|
| 游客 | 能看，不能写 |
| 注册 | "我能发帖了" |
| 活跃 | 用着用着就有了 |
| 认证 | 收到通知"你可以写长文章了" |
| 资深 | 收到通知"有个空间没人维护了，你想接手吗？" |

### 认领

当某个内容空间的作者长期不活跃（90 天无更新、无回应反馈），系统通知资深用户："XX 空间已经一段时间没人维护了，你可能感兴趣。"

这是 Lv4 权限。获得条件：认证后持续贡献 ≥ 90 天。

## 十、内容活性

### 反馈回路

```
有人点"有帮助" → 确认时间刷新 → 排序权重增加 → 更多用户看到 → 更多确认
有人点"有变化" → 通知作者 → 作者更新 → 信息重新变准 → 可信度恢复
长期无人确认 → 系统私下提醒作者 → 不公开，不评价
```

公理四·量变质变：每次"有帮助"是量变。100 次"有帮助"是质变——这篇内容成为该话题下的权威答案。

### 信息时效性

系统记录每条信息的：
- 上次确认时间
- 确认人数
- 反馈"有变化"的次数
- 变化频率（多快过时一次）

高频变化的信息（如食堂每日菜单）用短内容承载。低频变化的信息（如选课指南）用长文章承载。系统根据数据自动建议作者选择合适的形态。

## 十·附 需求回路

### 核心问题

微信群每年重复回答同样的问题，因为没人知道"这个问题被问了多少次"，也没有人把答案固定下来。

盘根的优势不只是"有答案"，而是**系统知道什么问题被反复问、什么信息缺失、什么内容需要更新**。用数据驱动内容生产，让同一个问题只需要被回答一次。

### 数据采集

系统静默记录以下行为数据（不追踪个人，只记录聚合模式）：

| 数据 | 来源 | 意义 |
|------|------|------|
| 搜索 query + 结果数 | 搜索日志 | 结果数 = 0 说明内容缺失 |
| 搜索 query + 点击 | 搜索日志 | 用户搜了但没点 = 结果不匹配 |
| AI 被调用的 query | AI 网关日志 | 高频 AI 问答 = 内容空白 |
| 帖子标签分布 | 内容分析 | 哪类话题最活跃 |
| 阅读完成率 | 阅读行为 | 读完 = 内容质量高 |
| "有变化"反馈频率 | 反馈表 | 高频变化 = 信息不稳定 |
| 空间访问量 | 空间访问日志 | 高访问 = 高需求 |

### 分析机制

公理四·量变质变：单个搜索是量变。同一个 query 被搜索 50 次是质变——这是明确的、可量化的人群需求。

```
系统每天运行分析任务：

1. 内容空白检测
   → 过去 30 天搜索量 > 20 次且无本地结果的 query
   → 标记为"高需求空白"
   → 示例："打印店在哪" 被搜了 45 次，0 篇相关内容

2. 内容质量预警
   → 阅读完成率 < 20% 的文章
   → "有变化"反馈 > 3 次的文章
   → 标记为"需要更新"

3. 需求趋势
   → 搜索量周环比增长 > 50% 的 query
   → 示例："考研报名" 搜索量本周增长 300%
   → 说明季节性需求来了，相关内容应该置顶

4. 供给缺口
   → 某空间访问量高但文章数 < 3
   → 说明有需求但没有足够的深度内容
```

### 闭环：从数据到行动

分析结果不躺在数据库里。它驱动三个具体的行动：

**行动 1：通知作者**

```
系统检测到"三食堂"空间近期访问量激增
  → 通知空间维护者：
    "最近很多同学在关注三食堂，你的空间本周被访问了 234 次。
     有 3 个高频搜索没有结果：'三食堂几点关门'、'三食堂外卖'、'三食堂推荐'。
     要不要补充这些内容？"
```

**行动 2：通知团队**

```
系统检测到内容空白
  → 团队后台显示：
    "过去 30 天有 45 次搜索无结果：'打印店在哪'。
     建议创建相关内容。"
  → 团队决定：自己写，或者邀请合适的认证作者写
```

**行动 3：优化搜索体验**

```
系统检测到高频 AI 问答
  → "选课怎么选" 过去一个月被问了 80 次，每次都是 AI 回答
  → 说明这个需求真实存在，但没有本地内容
  → 同时：搜索这个 query 时，优先展示"我知道准确信息，来补充"按钮
  → 引导用户把知识沉淀下来
```

### 消除重复

完整的消除重复流程：

```
第 1 次有人搜"打印店在哪" → 无结果 → AI 回答 → 搜索日志记录
第 5 次 → 系统标记为低频空白
第 20 次 → 系统标记为高需求空白 → 通知团队
          → 团队邀请作者写"校园打印指南"
          → 文章发布，归属"生活服务"空间
第 21 次有人搜"打印店在哪" → 本地有结果 → 展示文章 → 确认开始积累
第 50 次 → 23 人确认有帮助 → 这篇文章成为权威答案
从此：没有人需要再问"打印店在哪"了
```

**这就是盘根和微信群的终极差异。微信群里第 100 个人问同样的问题，和第 1 个人问的时候一样——没人知道这个问题被问了 100 次。盘根知道，并且自动把答案补上。**

### 年度循环

每年九月新生入学，同样的需求会重复出现。盘根的优势：

```
2025 年 9 月：新生搜"报到" → 系统有去年写的报到指南 → 内容被阅读、确认
                                              → 有人反馈"流程变了" → 作者更新

2026 年 9 月：新生搜"报到" → 系统有更新过的报到指南 → 不需要重新回答
```

每一年，知识积累得更多。每一年，重复提问更少。最终状态：新生打开盘根，搜什么都有，而且信息是最新的。

这就是公理二·熵减的终极形态：**校园信息从每年的混乱（新生不知道、重复问），变成永久的有序（有体系、有人维护、自动更新）。**

## 十一、AI 的角色

AI 是空气。无处不在，但不可见。

盘根的 AI 不是功能，是服务。学生不需要知道 AI 在帮他，就像用手机打电话的人不需要知道信号是怎么传输的。他只需要知道"搜了就有答案"。

### AI 在盘根中做的六件事

```
1. 搜索兜底 — 本地没答案时，AI 先答，保障每个问题都有回应
2. 情境计算 — 搜索框下方的 4 张推荐卡片，基于时间、历史、时效性实时计算
3. 内容组织 — 用户发帖时自动分类空间、推荐标签
4. 写作辅助 — 认证用户写文章时，AI 通过对话帮助组织思路
5. 需求发现 — 分析搜索和阅读数据，发现内容空白和需求趋势
6. 摘要生成 — 为长文章自动生成摘要，帮助读者快速判断是否需要深入阅读
```

学生感知到的是：搜索好用、信息准、写东西有人帮、首页信息总是相关的。
学生不需要知道的是：背后有 6 个 AI 能力在协同工作。

### "由 AI 生成"标签的策略

法律要求 AI 生成内容必须标注。但标注方式影响用户体验。

```
AI 回答卡片上：
  ┌──────────────────────────────┐
  │  三食堂有 8 个窗口...         │
  │  4 号窗口排队最短...          │
  │                              │
  │  由 AI 生成，仅供参考         │  ← 小字，不抢注意力
  │  [我知道准确信息，来补充]     │  ← 引导人类覆盖
  └──────────────────────────────┘
```

标注存在，但不是重点。重点是内容本身。标注的目的是合规 + 引导人类补充，不是让学生觉得"哦这是 AI 写的，不可信"。

### AI 服务人的边界

| AI 做什么 | AI 不做什么 |
|-----------|-----------|
| 找到相关信息并呈现 | 替用户决定信什么 |
| 在没有答案时给出参考 | 代替人类写内容 |
| 追问帮用户想清楚要说什么 | 决定什么重要什么不重要 |
| 排版和格式化 | 决定内容的最终形态 |
| 实时计算用户当下最需要的信息 | 追踪用户、推荐"你可能喜欢" |
| 监控信息时效性 | 替作者决定什么时候更新 |
| 分析搜索和阅读数据，发现内容空白和需求趋势 | 替团队决定做什么内容（只提供数据，决策权在人类） |

所有 AI 生成内容必须标注"由 AI 生成，仅供参考"。这是法律要求，也是信任要求。

## 十四、后端

后端与前端同步建设。当前本地开发以真实 `/api/*` 为主路径，前端通过 Vite proxy 连接后端；如需连接非默认后端，使用 `VITE_API_BASE_URL` 指定。

### 数据模型

```
全局字段：所有表含 site [cn|com]，原型期逻辑隔离，上线后物理隔离
用户（id, 手机号, 昵称, 学校, 等级, 创建时间, 最后活跃时间）
空间（id, 名称, 描述, 图标, 创建者id, 创建时间）
内容（id, 空间id, 作者id, 标题, 正文, 类型[帖子/文章], 标签,
      确认数, 最后确认时间, 浏览数, 收藏数, 变化标记, 创建时间, 更新时间）
反馈（id, 内容id, 用户id, 类型[有帮助/有变化], 反馈内容, 时间）
回复（id, 帖子id, 作者id, 正文, 星标数, 创建时间）
收藏（id, 用户id, 内容id, 时间）
搜索日志（query, 结果数, 是否AI回答, 时间）— 初期简单四字段，DAU > 100 后扩展为 search_events（含点击、停留、确认）
通知（id, 用户id, 类型, 标题, 内容, 关联id, 已读, 创建时间）
配置（key, value）— 存储认证条件阈值等可配置参数
```

### 技术栈

| 层级 | 技术 |
|------|------|
| API | Hono |
| 数据库 | PostgreSQL + Drizzle ORM |
| 认证 | 手机号验证码 + JWT |
| AI 网关 | 后端代理智谱 GLM-4-Flash |
| 文件存储 | 本地磁盘 / OSS（上线后） |
| 全文搜索 | PostgreSQL tsvector |

## 十七、不做的事

### 产品边界

- 不做评论区 → 文章用"有帮助/有变化"，帖子用回复
- 不做点赞 → 用"有帮助"确认代替
- 不做粉丝/关注
- 不做算法推荐 → 首页是活动排序的时间线 + 搜索；搜索框下方的情境信息不算推荐，是时效性提示
- 不做泛化内容 → 只做校园区域内容
- 不做浮动按钮 → 创建入口在首页内联
- 不做 Tab 切换（文章/讨论）→ 同一个流
- 不做颜色标签（绿黄红）→ 用时间透明代替

### 战略红线

| 红线 | 原因 |
|------|------|
| 不做流量生意 | 排序从准确度变成热度，作者会追流量而不是追准确 |
| 不开放空间创建权限 | 去掉门槛，质量崩塌，读者信任断裂。初期仅团队创建；达到一定规模后，资深用户可通过信任等级解锁创建权限（见第二十六节"创作权限"） |
| 不做以停留时长为目标的推荐 | 算法优化停留时长，盘根要的是"找到答案然后走"。搜索框下方基于时间的情境提示不算推荐 |
| 不为了增长降低标准 | 5 个高质量空间 > 50 个半成品 |
| 不让商业伤害信任 | 商家内容必须标明身份 |
| 不让用户感知 AI 的存在 | AI 是工具，不是角色 |

## 十八、成功指标

上线时必须能回答这五个问题：

| # | 问题 | 采集方式 | 目标 |
|---|------|---------|------|
| 1 | 有多少人在用？ | DAU | 100+ |
| 2 | 用户来做什么？ | 页面访问分布 | 搜索 > 30% |
| 3 | 内容有人读吗？ | 文章阅读完成率 + 阅读时长 | 中位数 > 30s |
| 4 | 活性飞轮转了吗？ | "有帮助" + "有变化"每日总数 | > 10 次/天 |
| 5 | 有人从读到写吗？ | 注册→发帖→认证转化漏斗 | 有人写第二篇 |

**最核心的信号：有人主动写了第二篇内容。**

不是被邀请写的，不是被奖励写的，是他自己想写的。这说明产品活了。

### AI 渗透度指标

作为"推动校园 AI 化"的隐性任务，需要追踪 AI 服务的渗透情况：

| 指标 | 采集方式 | 意义 |
|------|---------|------|
| AI 搜索占比 | 搜索日志中"是否AI回答"字段 | 有多少问题依赖 AI 兜底 |
| AI 回答后人类补充率 | AI 回答被人类内容替换的比例 | AI 在驱动内容生产 |
| 自动分类准确率 | 用户修改系统推荐空间的比例 | AI 理解用户意图的能力 |
| 全球站跳转率 | 从校园站点击进入全球站的比例 | 从"用 AI"到"学 AI"的转化 |

这些指标不用于对外展示，仅供团队判断 AI 服务是否真正在帮助学生。

## 二十二、冷启动

PRD 描述的是产品运行起来的理想状态。但产品上线第一天，没有用户、没有内容、没有数据。这一节回答：**从 0 到 1 怎么走。**

### 上线第一天

用户打开盘根，看到什么？

```
搜索框（正常工作，可以搜到种子内容）
搜索框下方 4 张推荐卡片：编辑精选（冷启动期，无用户行为数据，由团队手动推荐）
空间：种子内容已创建 3-5 个空间，每个空间有 3-5 篇文章
```

学生搜索"食堂" → 找到种子文章 → 读完 → 点"有帮助" → 第一次确认发生。

**上线第一天最重要的指标：有学生点"有帮助"。** 这说明他读到了内容，而且觉得有用。

### 上线第一周

- 10-20 个学生注册（线下推广：扫楼、食堂门口、宿舍）
- 每天有 5-10 次搜索
- 有人发第一条帖子
- 团队每天检查新发内容，人工审核（敏感词自动过滤已就绪，人工仅处理语义层面）

### 上线第一个月

- DAU 50-100
- 10-20 篇用户原创帖子
- 推荐卡片开始有数据（可以从编辑精选切换为 AI 推荐）
- 活动排序开始工作（有阅读、确认、收藏数据）
- 第一个用户满足认证条件，收到通知

### 上线前三个月

- 种子内容 50 篇全部就位（可以分批，不要求上线前全部完成）
- 上线时有 20 篇即可，后续持续补充
- 每个核心空间（报到、食堂、选课、生活）至少有 3 篇深度文章

### 推广路径

```
第一步：团队内部测试（1 周）→ 确保基本功能可用
第二步：邀请 10 个朋友试用（1 周）→ 收集反馈，修复问题
第三步：扫楼推广（食堂门口、宿舍楼下）→ 扫码注册
第四步：内容裂变（一篇好文章被分享到班级群）→ 自然增长
```

### 冷启动激励

PRD 删除了点赞和粉丝机制以避免流量导向，但冷启动阶段普通学生缺少创作动力。以下机制在冷启动期辅助数据积累，不与"做减法"公理冲突——它们不是长期产品功能，是阶段性补充：

**个人贡献可见化（所有用户）**

个人页展示贡献数据：发帖数、收藏数、帮助人数。让每个用户感受到"我做的事被看见了"。

贡献值计算：发帖 +10、被收藏 +5、被确认有帮助 +3。数值不在前端展示，但影响信任等级的自然升级。

**首次使用引导**

新用户首次打开时，首页顶部显示一条轻量引导条（可关闭，关闭后不再出现）：

```
"试着搜一个你关心的问题，比如'食堂几点关门'"   [×]
```

引导只出现一次，不强制。目的是降低首次使用门槛——让用户在 10 秒内完成第一个有价值的动作。

**线下活动绑定**

上线初期通过线下活动带动注册和内容生产：

| 活动 | 触发 | 产出 |
|------|------|------|
| 新生报到季 | 扫码领报到指南 → 注册 | 种子用户 + 报到类内容 |
| 社团招新 | 社团在盘根发活动帖 | 活动类帖子 |
| 二手市集 | 线下跳蚤市场同步线上 | 二手类帖子 |

活动结束后撤掉线下入口，用户已沉淀在平台上。

## 二十三、通知系统

通知是连接系统和用户的桥梁。认证、反馈、认领、提醒都通过通知触达。

### 通知类型

| 类型 | 触发条件 | 示例 |

## 二十四、与全球站的关系

### 不是两个独立产品，是一条路

盘根校园站（xyzidea.cn）和盘根 AI 指南针（xyzidea.com）是同一个品牌下的两个产品，服务于同一个人的不同阶段：

```
校园站：学生在用 AI（无感）
  → 日常搜索、信息获取、内容发布
  → AI 在后台为每个学生工作
  → 学生感受到的是"好用"，不是"AI 好用"

  ↓ 有人好奇了

全球站：学生在学 AI（有感）
  → 主动学习 AI 工具、技能、工作流
  → 从被动使用者变成主动掌握者
  → 探索更广阔的 AI 应用
```

**校园站是 AI 的大门。全球站是 AI 的深处。**

大多数学生停在门口就够了——他们只需要一个好用的工具。但总有学生会走进去——他们想理解背后的原理，想掌握更多的能力。全球站为他们而存在。

### 用户体验到的连接

连接不是强制跳转，不是横幅广告。是自然的、由好奇心驱动的：

```
学生在校园站搜索 → 看到 AI 回答卡片 → 底部"由 AI 生成，仅供参考"
  → 大部分人无视，继续用
  → 少部分人好奇："AI 还能做什么？"
    → 卡片旁边有一个小字链接："探索更多 AI 工具 →"
      → 点击进入全球站
```

不推。不催。等学生自己问。

### 转化触点设计

转化路径必须存在，但不主动推广。以下触点是学生在自然使用中可能接触到的：

| 触点 | 位置 | 形式 | 设计原则 |
|------|------|------|---------|
| AI 生成标签 | AI 回答卡片右上角 | Sparkles 图标 + "由 AI 生成" | 存在感低，不干扰阅读 |
| AI 工具探索链接 | AI 回答卡片底部 | 小字："想了解更多 AI 工具？" | 仅 AI 回答卡片出现，普通搜索结果不显示 |
| 全球站入口 | 文章页底部导航下方 | 一行文字："AI 还能做什么？看看全球站 →" | 在阅读完成后才出现，不打断主流程 |

三个触点的设计约束：
- 不在首页出现（首页是解决眼前问题的地方，不是导流的地方）
- 不在搜索过程中出现（搜索是获取信息的动作，不应该被打断）
- 颜色和字号降级（text-ink-faint，不突出）
- 新标签页打开，不离开校园站

### 工程层面的共享

| 共享内容 | 说明 |
|---------|------|
| @ns/shared | TypeScript 类型、敏感词库、AI 工具函数、API 契约 |
| 后端架构 | Hono + PostgreSQL + Drizzle。原型期共享数据库实例（`site` 字段隔离），上线后各自独立实例 |
| AI 网关 | 统一代理智谱 GLM-4-Flash，统一计费和限流 |
| 代码仓库 | 同一 monorepo，独立 package |

### 不共享的内容

| 维度 | 校园站 | 全球站 |
|------|--------|--------|
| 数据库 | 独立 | 独立 |
| 用户体系 | 独立 | 独立 |
| 服务器 | 境内 | 境外 |
| 内容 | 不互通 | 不互通 |
| 域名 | xyzidea.cn | xyzidea.com |

### 为什么这样设计

1. **法律要求**：cn 站数据不能出境，两站数据必须物理隔离
2. **产品逻辑**：校园生活和 AI 工具学习的用户场景不同，强融在一起会互相干扰
3. **信任模型**：校园站靠本地确认链，全球站靠平台审核，两者不能混用
4. **用户心智**：学生在校园站是"找信息"的心态，在全球站是"学东西"的心态

### 全球站定位更新

全球站 PRD 需要同步更新以下几点：

1. 明确全球站是"AI 主动学习平台"，不是"AI 内容展示站"
2. 校园站是用户获取的入口之一——用过校园站的学生，天然对 AI 有好感
3. Ferry 框架从"两站连接理论"调整为"团队产品哲学"，不是用户端概念
4. cn → com 内容单向同步——如果校园站产生了优质的 AI 使用经验帖，可以（经作者同意）同步到全球站作为案例

## 二十五、最终产品形态

> 本节描述盘根建成之后的完整形态。不是 V1，不是 MVP，是最终产品。
> 每一次迭代都朝这个方向走。来源：产品委员会第三次会议共识。
> **v9.0 更新**：底部 Tab 名称、日期/问候显示、搜索框 placeholder 以第二十六节为准。

### 一句话

**校园站：搜一下，可信的。**
**全球站：说一声，方案给你。**

两句话，两个产品，同一个使命——让 AI 服务于人。

### 一个场景

大一新生小林，九月到校报到。

**第一天。** 宿舍楼下扫码打开盘根。页面顶部搜索框"问点什么，或者分享点什么"。下方 4 张推荐卡片，"报到当天全流程"排在第一。点进去，文章底部写着"67 人确认有帮助"。照着走了一遍，顺利。搜"宿舍附近快递站在哪"，AI 回答了一张卡片，右上角小字"由 AI 生成"。没在意，去找快递站了。

**第一周。** 每天打开看一眼。推荐卡随时段变：早上的食堂、下午的图书馆、晚上的选课。搜"体育课推荐"，出来文章和回复，收藏了。发了第一条帖子："北苑宿舍水压好小，有人一样吗？"系统自动归到自由广场。第二天有人回复解决方案。

**第一个月。** 搜"挂科了怎么办"，没有本地内容，AI 回答了。收到通知："你已发布 3 条内容，获得 5 个确认。你的贡献正在帮助其他同学。"

**第三个月。** 写了文章"北苑生存指南"。有人反馈"有变化"——旁边新开了奶茶店。更新了文章。收到通知："你已满足认证作者条件，要不要解锁更多功能？"点了"好"。

**半年后。** 成为"北苑生存指南"空间维护者。文章被 200 人确认有帮助。某天搜东西时注意到 AI 回答卡片底部一行小字："想了解更多 AI 工具？"点了。新标签页打开全球站。

#### 搜索的最终形态

三种场景，三种处理方式。用户不需要知道背后的逻辑。

**场景一：本地有直接匹配**

用户搜"食堂几点关门" → 系统查到文章"三食堂测评"提到营业时间 → 返回文章卡片，高亮相关段落。AI 回答折叠在下方，用户可选择展开。

**场景二：本地无直接匹配**

用户搜"挂科了怎么办" → 本地无内容 → 调用 AI → 流式回答 → 卡片右上角标注"由 AI 生成"。下方推荐相关已有内容。

**场景三：本地有部分匹配**

用户搜"三食堂烤冷面好吃吗" → 找到帖子 + AI 回答 → 双结果展示：本地帖子（可信，有确认数）优先展示，AI 回答（补充）折叠在下方。

原则：**本地内容经过人工确认，可信度高于 AI 生成。AI 回答是补充，不是替代。**

#### 个人页

个人页不是设置页，是"我在盘根的身份"。

```
┌─────────────────────────────────────┐
│                                     │
│           (首字母头像)                │
│           小林                       │
│           黑河学院                   │
│                                     │
│  帮助了 23 人 · 450 阅读 · 12 收藏   │
│                                     │
│  ─────────────────────────────────  │
│  FileText 我的内容                   │
│     北苑生存指南（空间维护者）        │
│     水压好小有人一样吗？· 3条回复     │
│                                     │
│  Bookmark 我的收藏                   │
│     选课避雷指南                     │
│     三食堂测评                       │
│     打印店指南                       │
│                                     │
│─────────────────────────────────────│
│  [首页]    [探索]    [我的]           │
└─────────────────────────────────────┘
```

"帮助了 N 个人"的计算：
- 有人对我的内容点"有帮助" → 帮助 1 人
- 有人收藏我的内容 → 帮助 1 人
- 有人回复我的帖子并标记"解决了" → 帮助 1 人（"解决了"标记仅限 #求助 标签的帖子，由提问者操作）

加总即为帮助人数。这个指标比粉丝数更有意义——它衡量实际影响，不是注意力。指标决定行为。

#### AI 的角色边界

AI 做的事：

| 场景 | AI 行为 | 用户感知 |
|------|---------|---------|
| 搜索无结果 | AI 生成回答 | "搜了就有答案" |
| 长文章 | AI 生成摘要 | "不用读完就知道大意" |
| 写文章 | AI 对话引导 | "有人帮我理清思路" |
| 过期内容 | AI 检测信息可能过期 | "这篇文章可能需要更新" |
| 搜索日志 | AI 分析高频无结果 query | 团队知道该写什么 |

AI 不做的事：

| 禁止行为 | 原因 |
|---------|------|
| 不自动发布内容 | 人类主权：AI 不替人决定什么该发 |
| 不自动精选/推荐 | 做减法：推荐是编辑的事 |
| 不替用户做选择 | 人类主权：AI 提供信息，人决定 |
| 不展示 AI 头像/助手界面 | AI 是空气，不是角色 |
| 不在没有本地内容时编造事实 | 冷启动诚实策略 |

AI 做三件事——回答、整理、引导。不做一件事——替人决定。

### 全球站最终形态

校园站的学生点了全球站链接后，看到的不是工具列表或教程目录，是一个更简单的东西：

```
"你想用 AI 做什么？"
  → 写东西
  → 做视频
  → 写代码
  → 提高效率

选一个 → AI 帮你生成一个具体方案 → 你可以直接用
```

全球站的核心体验是**方案生成**。用户不需要理解 AI 工具生态，不需要知道不同模型的区别。只需要说"我想用 AI 做 X"，系统给出一个可以立即执行的方案。

#### 搜索结果页

三种结果模式（不暴露技术概念）：
1. 本地精确匹配：展示文章卡片，高亮相关段落，AI 回答折叠在下方
2. 本地无匹配：AI 流式回答，标注"由 AI 生成，仅供参考"，底部"我知道准确信息，来补充"
3. 部分匹配：本地内容优先（更高信任），AI 回答折叠在下方

- 知识区：系统性文章，大卡片（文章嵌套深度最多两层）
- 动态区：帖子（小卡片）+ 变动提醒

- 桌面端三栏：左文章树 | 中内容区 | 右目录（长文章时）
- 移动端单栏

#### 文章阅读页

- 桌面端三栏：左空间文章树 | 中正文 + 目录 | 右侧信息

- 底部两个按钮：**有帮助** / **有变化**
- "有变化"是公开信息，显示在文章标题下方

- 我的内容（帖子、文章）

- 设置

### 创作权限（按信任等级解锁）

| 用户状态 | 能做什么 | 入口在哪 |
|---------|---------|---------|
| 访客 | 搜索、浏览 | 无创作入口 |
| 注册用户 | 发短帖 | 空间页底部输入框 + 首页"写点什么"。未选择空间时默认归属"自由广场" |
| 认证用户 | 写长文章（AI 对话引导） | 收到通知后，导航出现"写"入口 |
| 资深用户 | 创建空间、认领空间 | 个人页出现"创建空间"按钮 |

创作能力不是 UI 上的固定元素。用户通过行为自然积累信任，系统推送通知解锁新能力。用户感知到的是自然事件，不是等级系统。

### 信任等级（内部，用户不可见）

等级在 UI 中完全隐藏。用户看到的是贡献数据和自然通知。初期只激活 3 个等级（标注 ★），其余等运营需要时激活。

| 等级（内部） | 解锁能力 | 用户感知 | 初期状态 |
|-------------|---------|---------|---------|
| Visitor | 搜索、浏览 | 无感知 | ★ 活跃 |
| Registered | 发短帖 | "我可以发帖了" | ★ 活跃 |
| Active | 更多发帖权限 | 无感知（自然过渡） | 暂未激活 |
| Authenticated | 写长文章 + 个人空间 | 收到通知："你可以写长文章了" | 暂未激活 |
| Senior | 创建空间 + 认领空间 | 收到通知："你可以创建空间了" | 暂未激活 |
| Admin | 后台管理 | 内部角色 | ★ 活跃 |

### 数据诚实原则

冷启动阶段所有数据必须真实：
- 确认数从 0 开始，不造假
- 推荐卡片展示真实的空间和文章
- 浏览量、收藏数等均为真实用户行为产生

**学生信不信盘根，取决于第一次搜索体验。** 假数据会破坏信任。

### MVP 范围："一个食堂"计划

先跑通核心循环，再扩展。

**核心循环：**
```
学生搜索 → 命中本地内容 → 看到文章 → 点"有帮助" → 确认数+1
```

**MVP 包含（三周结束时）：**

| 页面 | 功能 |
|------|------|
| 首页 | 搜索框 + 4 张推荐卡片（第三周加入） |
| 搜索结果 | 本地匹配 + AI 兜底（第二周开启 AI） |
| 文章阅读 | 正文 + "有帮助"按钮 |
| 空间页 | 一个空间下的所有文章 |

**MVP 内容范围：** 3-5 个高频话题（食堂、报到、选课），20-30 篇团队手写的真实文章。

**MVP 暂不做：** 信息流、帖子系统、创作入口、通知系统、信任等级、个人页、空间目录、桌面端深度适配。

**核心指标：** 有没有一个学生搜了问题，点了"有帮助"。

### 开发优先级

移动端优先。学生在食堂用手机搜"这个窗口好吃吗"是真实场景。坐在电脑前搜食堂信息不是。等有人开始写长文章了，再做桌面端深度适配。

### 补充说明

#### 帖子详情

帖子不设独立页面。通过信息流或空间页的预览 overlay 展示完整内容和回复。overlay 内可直接回复，"查看全部回复"仍在 overlay 内展开。降低跳转成本。

#### 写作页面

AI 对话式写作不作为独立页面。从空间页底部输入框或首页"写点什么"进入。长内容触发 AI 对话引导流程，以 overlay 或内嵌面板形式展示，不离开当前页面上下文。

#### 通知系统

通知 UI 位置：
- 桌面端：顶栏头像旁铃铛图标，下拉展示通知列表
- 移动端："我的"页面顶部通知入口

6 种通知类型（见第二十三节）：认证邀请、内容反馈、变动提醒、过期提醒、空间认领邀请、帖子回复。
